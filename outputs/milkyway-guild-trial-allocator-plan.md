# Milky Way Idle 公会试炼自动分配插件方案

## 目标

做一个 Tampermonkey 油猴插件，在 Milky Way Idle 公会页面读取当周 4 个生活试炼和 2 个战斗试炼，结合成员资料生成报名推荐表。

插件只做本地辅助推荐和复制名单，不直接报名、不调用隐藏接口、不绕过游戏权限。

## 资料和假设

- 已核对公开信息：公会试炼每周随机出现 4 个生活试炼和 2 个战斗试炼，战斗遭遇包括 Badger、Chameleon、Jellyfish、Hedgehog、Swarm。
- 公开资料没有稳定给出五个试炼 Boss 的完整数值表；本周獾和虫群使用用户提供的 Lv.100 游戏内属性截图校准，其余 Boss 仍使用可编辑经验权重。
- 插件按 Boss 倾向匹配物理、法系、AOE、攻击流派、坦克、治疗、续航、辅助等成员评分。
- 插件只读监听游戏已经接收的 `guild_characters_updated` 消息，不主动请求服务器。消息中的 `guildTrialSignupLevelMap` 可提供已报名成员的试炼快照等级。
- 如果公会实战发现某个首领更适合其他配置，可以在“高级维护”中调整首领属性权重。

## 公会试炼规则口径

- 公会周于每周五 `00:00 UTC` 重置，每周随机获得 4 场生活试炼和 2 场战斗试炼。
- 每名成员每周最多报名 1 场生活试炼和 1 场战斗试炼。
- 生活试炼先进行，战斗试炼随后进行；每场本周最多持续 1 小时。
- 试炼与角色正常行动并行，使用所选试炼配装的快照，不使用任何消耗品。
- 试炼从 Lv.100 开始，每层提高 10 级，最高 Lv.300。
- 战斗遭遇为獾、变色龙、水母、刺猬、虫群五种，每周随机出现其中两种，怪物 HP 随参与人数增加。
- 生活营地和战斗营地决定报名名额；17 座试炼建筑以及神龛增益会影响成员在试炼中的实际属性。
- 分配评分必须按“试炼配装快照、无消耗品、包含当前公会建筑和神龛增益”的口径填写。
- 通关层数决定公会点数和参与者公会代币，并影响试炼结束后的全公会额外代币奖励。

## 生活试炼规则

1. 对每个成员读取当周生活试炼对应评分。
2. 把每场试炼的总名额作为容量约束，每名成员最多选择一场。
3. 按成员的专业优势差和最高分排序，再使用贪心近似 Leximin 放置：每次选择能使“从最弱组开始排列的组强度向量”最大的试炼。
4. 组强度按 `sum(score) / (1 + 0.01 * 人数)` 折算，把每名参与者使共享池需求增加 1% 纳入均衡判断。
5. 开启“重新规划全部报名”后，当前已报名人数不占用最终容量，只用于输出保持、改报和取消名单。
6. 缺少全部生活评分的成员只会在剩余名额中按名额压力兜底，不参与有评分成员的 Leximin 判断。

自动读取的生活评分是专业等级，因此默认结果是“按等级代理的均衡方案”，不是精确工作量最优。若 CSV 填每小时有效工作量，折算组强度才代表有效工作量。当前实现是浏览器内可快速运行的贪心近似 Leximin，不保证枚举所有组合后的数学全局最优。

## 生活成功率模型

可以继续计算前 K 个高期望方案和逐层成功率，但需要以下输入，不能只依靠等级：

- 每名成员在每场生活试炼配装快照下的一小时工作量期望 `μ`。
- 对应工作量方差 `σ²`，或足以推导方差的行动时间、单次工作量和随机触发概率。
- Lv.100 至 Lv.300 每层共享工作池基础需求 `W(level)`。
- 每名参与者使需求增加 1% 的精确计算顺序和取整规则。

若各成员工作量近似独立，某层成功率可用 `P(sum(X_i) >= W(level) * (1 + 0.01n))` 计算；数据齐全后可用动态规划/束搜索输出前 K 个方案、期望通关层数及每层成功率。当前插件不在缺少这些数据时伪造概率。

## 战斗试炼规则

战斗不再只按“变色龙 + 刺猬”写死，而是对每周出现的两个 Boss 分别打分：

1. 读取 Boss 默认属性权重。
2. 读取成员资料中的战斗评分字段。
3. 按完整 Boss 权重计算适配分，再混入 20% 通用战斗评分，避免通用战力完全掩盖属性克制。
4. 先放置填写了 `fixedCombat` 的固定成员；固定位置是硬约束，不受 `avoid` 和边际门槛影响。
5. 其余成员先按“对某个 Boss 的适配优势差”排序，再用贪心近似 Leximin 平衡两个战斗组的折算强度。
6. 每增加一名非固定参与者，按怪物 HP 增加 1% 计算边际门槛；只有成员适配分足以抵消新增 HP 时才加入。
7. 没有战斗评分的非固定成员不自动塞入战斗试炼，避免只增加怪物 HP 而无法确认有效贡献。

默认 Boss 倾向：

- `badger`：本周截图中魔法闪避 `374`，低于三种近战闪避 `517` 和远程闪避 `660`，优先 `magic` + `single`；其攻击为斩击物理，仍需要坦克和续航。
- `chameleon`：物理单体，吃 `physical`、`single`。
- `jellyfish`：续航/辅助/法系，吃 `magic`、`healer`、`sustain`、`support`。
- `hedgehog`：法系单体，吃 `magic`、`single`、一定续航。
- `swarm`：四目标混编，第一优先级为 `aoe`；按四只怪的平均闪避，法系整体略占优，其次是钝击和远程，同时提高 `tank`、`healer`、`sustain`、`support` 权重。

本周虫群 Lv.100 截图依据：

- 甲虫：钝击物理；钝击/魔法闪避 `429`，其他主要闪避 `550`。
- 蜻蜓：远程物理；三种近战闪避 `352`，远程 `473`，魔法 `616`。
- 黄蜂：刺击物理；魔法闪避 `352`，三种近战 `473`，远程 `605`。
- 萤火虫：火系魔法；远程闪避 `352`，魔法 `473`，三种近战 `660`。

这些数值用于判断相对弱点，不把 Lv.100 的绝对数值直接当成高层数的固定属性。

## CSV 字段

必填：

- `name`：玩家名。

生活评分：

- `milking`
- `foraging`
- `woodcutting`
- `cheesesmithing`
- `crafting`
- `tailoring`
- `cooking`
- `brewing`
- `alchemy`
- `enhancing`

战斗评分：

- `combat`：通用战斗评分。
- `single`：单体能力。
- `aoe`：虫群/AOE 能力。
- `physical`：物理输出能力。
- `magic`：法系输出能力。
- `stab`：刺击输出能力，可选。
- `slash`：斩击输出能力，可选。
- `blunt`：钝击输出能力，可选。
- `ranged`：远程输出能力，可选。
- `tank`：坦克能力。
- `healer`：治疗能力。
- `sustain`：续航、生存、稳定性。
- `support`：光环、复活、易伤、群疗、喷泉等团队辅助能力。
- `power`：备用战力评分。
- `level`：游戏公开的通用角色等级（如果公会数据包含）。
- `lifeLevel`：已报名生活试炼的快照等级。
- `combatLevel`：已报名战斗试炼的快照等级，同时在 `combat` 为空时自动填入通用战斗评分。
- `currentLife`：当前生活报名方向，只用于生成保持、改报、取消名单。
- `currentCombat`：当前战斗报名方向，只用于生成保持、改报、取消名单。
- `role`：`dps`、`tank`、`healer`、`debuff`，也支持中文“输出/坦克/治疗/减益”。
- `damageType`：`physical` 或 `magic`，也支持中文“物理/法系/魔法”。
- `fixedCombat`：固定战斗试炼，可填当周 Boss 的英文 key 或中文名。填写后先占对应名额；目标本周未出现或名额不足时保持未分配，不改派其他场。

角色战斗导出还会增加原始或摘要字段：

- `attackLevel`、`defenseLevel`、`meleeLevel`、`rangedLevel`、`magicLevel`、`staminaLevel`、`intelligenceLevel`：导出的七项战斗技能等级。
- `weaponType`：从主手/副手识别的武器类别；用于填写 `damageType` 及对应的 `magic`、`physical`、`ranged`、`stab`、`slash`、`blunt` 等等级代理分。
- `equipmentCount`、`combatEquipmentCount`、`maxEnhancement`：装备摘要；生产工具不进入战斗装备数。
- `abilityCount`、`maxAbilityLevel`：能力摘要。
- `combatHouseLevelSum`：餐厅、图书馆、道场、健身房、军械库、射箭场、神秘书房的等级和，仅作核对字段，不直接当作战力。

能力代理分只采用导出中可验证的目标信息：目标为 `all_enemies` 的能力填入 `aoe`，`targeted_enemy` 填入 `single`，`all_allies` 填入 `healer`，光环、帷幕、复活类能力填入 `support`。这些数值仍是能力等级代理，不是完整战斗模拟结果。

可选：

- `preferLife`：生活偏好，会加权。
- `preferCombat`：战斗偏好，会加权。
- `avoid`：不想去的试炼，多个用逗号、分号或竖线分隔。

## 使用流程

1. 会长在“试炼”页点击“读取试炼”，确认识别到 4 场生活和 2 场战斗试炼。
2. 会长在“成员”页点击“读取成员”和“拉取会员资料”。
3. 只为减益、坦克等特殊成员选择职责、固定战斗或偏好；其他成员保持自动分配。
4. 需要推翻乱报名时，在“连接”页勾选“重新规划全部报名”。
5. 点击“生成分配”，检查分组、技能建议和未分配名单，然后点击“发布方案”。

原始表格、角色导出内容和首领权重只保留在“高级维护”中，正常流程无需编辑。

只导入姓名时，生活试炼可以按剩余名额兜底；战斗试炼因参与人数会提高怪物 HP，不会给无战斗评分成员自动报名。补充生活专业和战斗评分后，同一成员会自动改用属性匹配结果。

## 会员端与远程汇总

远程模式由三个交付物组成：

- `milkyway-guild-trial-member.user.js`：会员读取和上传自己的裁剪后资料，查看个人适配排名和正式分配。
- `milkyway-guild-trial-allocator.user.js`：会长同步本周会员名单、拉取会员资料、运行全局分配算法并发布方案。
- `remote-worker/`：Vercel 同步服务，通过 fine-grained PAT 把明文 JSON 保存到私有仓库。

远程流程：

1. 会长部署同步服务，设置 GitHub 凭据和会长令牌。
2. 会长读取成员，并在“连接”页点击“同步本周名单”，发布本周项目、首领权重和成员姓名。
3. 会员端填写相同公会编号，只在点击“一键上传并查看分配”时上传；不会后台自动上传。
4. Worker 按本周名单和当前角色名匹配成员，并以规范化姓名生成稳定成员 ID。
5. 会长拉取本周上传后，数据合并进成员资料，但不覆盖同名成员已有的手工评分。
6. 会长生成全局方案并发布。服务按公会编号和角色名返回对应成员的生活、战斗分配和技能建议。

会员端显示两种结果：个人适配排名由本人属性和本周权重立即计算；正式分配必须等待会长汇总全体成员并发布。两者可能不同，因为全局方案还要考虑名额、其他成员的相对优势和战斗人数带来的怪物 HP 增长。

仓库文件使用普通 JSON，仓库必须保持私有。GitHub fine-grained PAT 只存在于 Vercel 环境变量中，不进入任何油猴脚本。详细部署步骤见 `remote-worker/README.md`。

## 当前限制

- 试炼名称和报名人数从每个可见试炼卡片的最小独立 DOM 容器中成对读取，不再按整页文本向后匹配。状态栏必须显示“生活 4/4、战斗 2/2”才允许生成。
- 网页 DOM 改版时，自动读取试炼卡片和成员表可能失效；数据不完整时插件会停止生成，避免静默使用错误数据。
- 公会成员表本身不包含完整公开资料。插件只能在用户正常打开成员资料后读取可见内容；当前支持“概览”的总等级、战斗等级、战力评分，以及“专业”页按固定顺序显示的10项生活等级。
- 完整角色导出只适用于导出者本人；公开成员资料消息不包含他人的完整装备、能力和房屋配置。因此每名成员仍需自行导出并交给会长导入，插件不会后台批量请求。
- 角色导出中的食物、饮料和物品消耗触发会被丢弃，因为公会试炼不使用消耗品。装备仅用于识别武器类别并保存摘要，房屋仅保存等级摘要；缺少官方属性公式时不会把装备价格、强化等级或房屋等级直接伪造成战力。
- 导出的技能和能力等级会作为 Boss 属性匹配的代理分，但并不包含命中、闪避、伤害公式、装备词条、公会建筑和神龛的完整计算，不能等同于模拟器胜率。
- 未报名成员通常不在 `guildTrialSignupLevelMap` 中，因此无法自动获得完整专业等级；插件不会为此逐个请求成员资料，以遵守游戏对脚本不得主动发送服务器请求的限制。
- Boss 权重是可调经验规则，不是官方完整战斗模拟。
- 远程服务必须由会长自行部署；交付代码不包含用户的 GitHub PAT 或 Vercel 账号凭据。
- 姓名匹配不是强身份认证。懂接口的人可以伪造名单中的角色名并覆盖资料或读取该角色分配；需要防冒用时必须恢复签名令牌或接入官方可验证身份。
